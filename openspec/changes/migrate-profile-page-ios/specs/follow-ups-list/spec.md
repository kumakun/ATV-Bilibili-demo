## 新增需求

### 需求：显示关注列表
系统必须以列表形式显示用户关注的所有 UP 主。每个 UP 主必须显示头像、用户名和个人签名。

#### 场景：成功加载关注列表
- **当** 用户打开关注 UP 主列表页面
- **那么** 系统必须从服务器获取关注列表数据
- **并且** 显示包含头像、用户名和签名的 UP 主卡片

#### 场景：加载失败
- **当** 获取关注列表失败
- **那么** 系统必须显示错误提示信息
- **并且** 允许用户重试

#### 场景：空列表
- **当** 用户没有关注任何 UP 主
- **那么** 系统必须显示空状态提示

### 需求：分页加载
系统必须支持分页加载关注列表，每页加载固定数量的 UP 主数据。

#### 场景：首次加载
- **当** 用户打开关注列表
- **那么** 系统必须加载第一页数据（最多 40 个 UP 主）

#### 场景：滚动加载更多
- **当** 用户滚动到列表底部
- **并且** 还有更多数据未加载
- **那么** 系统必须自动加载下一页数据

#### 场景：已加载全部数据
- **当** 用户滚动到列表底部
- **并且** 已加载所有关注的 UP 主
- **那么** 系统禁止继续请求更多数据

### 需求：防止重复请求
系统必须防止在上一次请求未完成时发起新的数据请求。

#### 场景：请求进行中
- **当** 系统正在加载某一页数据
- **并且** 用户触发加载更多操作
- **那么** 系统必须忽略新的请求
- **并且** 等待当前请求完成

### 需求：显示 UP 主头像
系统必须为每个 UP 主显示圆形头像，头像必须从网络加载并进行适当优化。

#### 场景：加载 UP 主头像
- **当** 显示 UP 主卡片
- **那么** 系统必须加载并显示该 UP 主的头像
- **并且** 头像必须以圆形形式显示
- **并且** 使用图片降采样优化内存

#### 场景：头像加载失败
- **当** UP 主头像加载失败
- **那么** 系统必须显示默认头像占位符

### 需求：显示 UP 主信息
系统必须显示 UP 主的用户名和个人签名。

#### 场景：显示完整信息
- **当** UP 主有用户名和签名
- **那么** 系统必须显示用户名在上方
- **并且** 显示个人签名在下方

#### 场景：签名过长
- **当** UP 主签名文本过长
- **那么** 系统必须截断或省略显示

### 需求：导航到 UP 主空间
系统必须支持用户点击 UP 主卡片后导航到该 UP 主的个人空间页面。

#### 场景：点击 UP 主卡片
- **当** 用户点击某个 UP 主的卡片
- **那么** 系统必须打开该 UP 主的个人空间页面
- **并且** 传递 UP 主的 mid 参数

### 需求：网格布局显示
系统必须使用网格布局展示关注列表，在 iOS 平台上优化为适合移动设备的布局。

#### 场景：在 iOS 上显示
- **当** 在 iOS 设备上显示关注列表
- **那么** 系统必须使用适合移动设备屏幕的列数
- **并且** 保证每个卡片有足够的空间显示内容
