## 新增需求

### 需求：系统必须以多列网格展示更新动态

系统必须将更新动态页面中的视频内容以多列网格方式展示，以提高浏览效率并充分利用横向空间。

#### 场景：在 iPhone 上显示更新动态

- **当** 用户在 iPhone 上打开更新页面
- **那么** 系统必须以 2 列网格展示视频动态卡片
- **那么** 系统必须保证每个卡片仍可显示封面、标题、UP 主信息和核心播放信息

#### 场景：在更宽环境中显示更新动态

- **当** 更新页面运行在 iPad 或其他更宽的可用宽度环境中
- **那么** 系统必须使用多于 2 列的网格展示视频动态卡片
- **那么** 系统必须根据可用宽度自适应增加列数
- **那么** 系统禁止退化为与 iPhone 相同的固定 2 列布局，除非可用宽度已缩小到无法满足卡片最小可读宽度

#### 场景：宽度变化时保持可读性

- **当** 设备方向、窗口尺寸或容器宽度发生变化
- **那么** 系统必须重新计算网格列数
- **那么** 系统必须优先保证卡片内容可读，而不是盲目增加列数
- **那么** 布局变化后分页加载、刷新和点击跳转行为必须保持正常

## 修改需求

### 需求：系统必须展示关注动态列表

系统必须在更新页面展示用户关注的 UP 主发布的视频动态列表，包括视频封面、标题、UP 主信息、发布时间和播放数据。

#### 场景：用户打开更新页面

- **当** 用户打开更新页面
- **那么** 系统必须自动加载并展示第一页的关注动态列表
- **那么** 每个动态卡片必须显示视频封面、标题、UP 主头像、UP 主名称、发布时间、播放量、弹幕数

#### 场景：没有关注动态

- **当** 用户没有关注任何 UP 主或关注的 UP 主没有发布动态
- **那么** 系统必须显示空状态提示："暂无关注动态"

#### 场景：加载失败

- **当** 动态数据加载失败（如网络错误）
- **那么** 系统必须显示错误提示信息
- **那么** 系统必须提供"重试"按钮允许用户重新加载
