## 上下文

应用之前已经在 UI 层面移除了直播、推荐、热门、排行榜功能的入口，主 Tab Bar 从 6 个标签页简化为 3 个（关注、收藏、我的）。但相关的代码文件仍然存在于代码库中，包括：
- 完整的直播模块（5 个文件）
- Feed 组件（4 个文件）
- 三个已废弃的 ViewController（FeedViewController、HotViewController、RankingViewController）

这些未使用的代码会增加代码库的维护负担，并可能导致混淆。现在需要彻底清理这些代码文件。

**当前状态：**
- `BLTabBarViewController` 已更新为仅包含 3 个标签页，并包含了处理旧索引映射的兼容逻辑
- 主导航规范已更新为描述 3 标签页结构
- UI 上已无法访问被移除的功能

**约束：**
- 必须保留现有的索引映射逻辑，确保从旧版本升级的用户不会遇到问题
- 删除文件后必须更新 Xcode 项目文件，避免构建错误
- 需要确保没有其他代码引用这些被删除的文件

## 目标 / 非目标

**目标：**
- 删除所有与直播、推荐、热门、排行榜相关的代码文件
- 更新 Xcode 项目文件以移除对已删除文件的引用
- 确保删除后应用仍能正常构建和运行
- 保持代码库整洁，降低维护成本

**非目标：**
- 不修改 `BLTabBarViewController` 的索引映射逻辑（已经正确实现）
- 不删除可能被其他模块共用的基础组件
- 不涉及任何新功能开发
- 不修改应用的架构或导航结构（已在之前的变更中完成）

## 决策

### 决策 1：直接删除文件而非标记为废弃

**选择方案：** 直接从代码库中删除文件

**理由：**
- 这些功能已在 UI 层面完全移除，用户无法访问
- 没有计划恢复这些功能
- 通过 Git 历史可以随时恢复代码（如果需要）
- 减少代码库大小和复杂度

**考虑的替代方案：**
- **保留文件但注释掉**：会导致代码库混乱，维护成本高
- **移动到 Archive 目录**：仍占用空间，且不如 Git 历史清晰

### 决策 2：分批删除文件

**选择方案：** 按模块分组删除

**分组策略：**
1. `BilibiliLive/Module/Live/` - 直播模块（5 个文件）
2. `BilibiliLive/Component/Feed/` - Feed 组件（4 个文件）
3. `BilibiliLive/Module/ViewController/` 中的 3 个独立 ViewController

**理由：**
- 按功能模块分组便于理解和追踪
- 如果出现问题，可以更容易定位是哪个模块导致的
- 便于 Code Review 和验证

### 决策 3：使用文件系统删除而非仅更新项目文件

**选择方案：** 同时删除物理文件和更新 Xcode 项目引用

**理由：**
- 确保文件从磁盘彻底删除
- Xcode 项目文件会自动更新（或需要手动更新）
- 避免"幽灵引用"问题

**执行方式：**
- 使用 `rm -rf` 命令删除目录
- 使用 `rm` 命令删除单个文件
- 在 Xcode 中或通过编辑 `.pbxproj` 文件更新项目引用

## 风险 / 权衡

### 风险 1：其他模块可能依赖被删除的代码
**缓解措施：**
- 在删除前使用 `grep` 搜索整个代码库，检查是否有其他文件引用这些即将删除的类或文件
- 特别检查 Storyboard 文件中是否有对这些 ViewController 的引用
- 删除后运行完整构建，确保没有编译错误

### 风险 2：Xcode 项目文件可能损坏
**缓解措施：**
- 在操作前提交所有更改
- 如果手动编辑 `.pbxproj`，先备份原文件
- 删除后在 Xcode 中打开项目，验证项目结构正确

### 风险 3：无法恢复功能
**权衡：**
- **优势**：简化代码库，降低维护成本
- **劣势**：如果未来需要恢复功能，需要从 Git 历史恢复
- **接受理由**：功能已决策移除，Git 提供了充分的恢复能力

### 风险 4：删除的组件可能有共享依赖
**缓解措施：**
- `BilibiliLive/Component/Feed/` 中的组件（如 `StandardVideoCollectionViewController`）可能被其他功能使用
- 在删除前检查这些类是否在其他地方被引用
- 如果发现共享使用，需要保留该文件或重构代码
