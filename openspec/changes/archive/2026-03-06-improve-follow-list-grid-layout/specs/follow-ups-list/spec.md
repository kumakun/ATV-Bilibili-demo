## 修改需求

### 需求：网格布局显示
系统必须使用网格布局展示关注列表，并且必须根据设备类型与可用宽度采用不同的列数策略，以保证浏览效率和卡片内容可读性。

#### 场景：在 iPhone 上显示关注列表
- **当** 在 iPhone 设备上显示关注列表
- **那么** 系统必须使用 2 列网格展示关注 UP 主卡片
- **并且** 系统必须保持每个卡片可显示头像、用户名和签名摘要

#### 场景：在 iPad 上显示关注列表
- **当** 在 iPad 设备上显示关注列表
- **那么** 系统必须使用多列网格展示关注 UP 主卡片
- **并且** 网格列数必须根据可用宽度自适应增加
- **并且** 系统禁止退化为与 iPhone 相同的 2 列移动端布局（除非窗口宽度已缩小到无法满足卡片最小可读宽度）

#### 场景：在 iPad 宽屏下提高内容密度
- **当** 关注列表在 iPad 的常规宽度或更宽布局中显示
- **那么** 系统必须在同一屏幕中展示多于 iPhone 2 列布局的卡片数量
- **并且** 保持卡片间距与内容排版稳定，不因增加列数造成主要信息不可读

#### 场景：窄宽度环境下保持可读性
- **当** 关注列表所在窗口宽度缩小（例如 iPad 分屏或尺寸变化）
- **那么** 系统必须重新计算网格列数
- **并且** 优先保证卡片内容可读，而不是盲目维持更高列数
- **并且** 布局调整后必须保持分页加载与点击导航行为正常
