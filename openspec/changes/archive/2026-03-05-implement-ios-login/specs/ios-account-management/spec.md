## 新增需求

### 需求：注册新账号

系统必须能够接收登录成功后的凭证，并创建或更新账号记录。

#### 场景：首次登录注册账号
- **当** 登录成功并获得 token 和 cookies
- **那么** 系统必须调用用户信息 API 获取用户资料（mid、用户名、头像）
- **并且** 系统必须创建包含 token、cookies 和用户资料的账号记录
- **并且** 系统必须将该账号设置为当前活跃账号
- **并且** 系统必须将账号数据持久化到本地存储

#### 场景：API 请求失败时使用默认资料
- **当** 用户信息 API 请求失败
- **那么** 系统必须使用 token 中的 mid 创建默认用户资料
- **并且** 用户名必须设置为 "UID {mid}"
- **并且** 头像必须设置为空字符串
- **并且** 系统必须继续完成账号注册流程

#### 场景：重复登录更新现有账号
- **当** 登录的账号已存在于本地存储中
- **那么** 系统必须更新现有账号的 token 和 cookies
- **并且** 系统必须更新最后活跃时间
- **并且** 系统必须将该账号设置为当前活跃账号

### 需求：管理活跃账号

系统必须维护一个当前活跃的账号状态。

#### 场景：设置活跃账号
- **当** 用户成功登录或切换账号
- **那么** 系统必须将指定的账号设置为活跃状态
- **并且** 系统必须更新该账号的最后活跃时间
- **并且** 系统必须应用该账号的 cookies 到 HTTP 客户端

#### 场景：获取当前活跃账号
- **当** 应用需要访问用户信息
- **那么** 系统必须返回当前活跃账号的完整信息
- **并且** 如果没有活跃账号，系统必须返回 nil

#### 场景：应用启动时自动激活账号
- **当** 应用启动时没有活跃账号，但存在已保存的账号
- **那么** 系统必须自动选择最后活跃时间最近的账号
- **并且** 系统必须将其设置为当前活跃账号
- **并且** 系统必须应用该账号的 cookies

### 需求：检查登录状态

系统必须提供快速检查用户登录状态的方法。

#### 场景：已登录状态
- **当** 存在活跃账号
- **那么** `isLoggedIn` 属性必须返回 true

#### 场景：未登录状态
- **当** 不存在活跃账号
- **那么** `isLoggedIn` 属性必须返回 false

### 需求：持久化存储账号数据

系统必须将账号信息安全地保存到本地存储。

#### 场景：保存账号列表
- **当** 账号数据发生变化（新增、更新、删除）
- **那么** 系统必须使用 JSONEncoder 编码所有账号
- **并且** 系统必须将编码后的数据保存到 UserDefaults 的 `app.multiple.accounts` 键

#### 场景：保存活跃账号 ID
- **当** 活跃账号发生变化
- **那么** 系统必须将活跃账号的 mid 保存到 UserDefaults 的 `app.multiple.accounts.active` 键

#### 场景：应用启动时加载账号数据
- **当** AccountManager 初始化
- **那么** 系统必须从 UserDefaults 加载账号列表
- **并且** 系统必须从 UserDefaults 加载活跃账号 ID
- **并且** 如果数据解码失败，系统必须使用空账号列表

### 需求：管理 Cookies

系统必须管理账号关联的 HTTP Cookies。

#### 场景：应用账号 Cookies
- **当** 设置活跃账号
- **那么** 系统必须将该账号的所有 cookies 应用到 HTTP 客户端
- **并且** 系统必须清除之前账号的 cookies

#### 场景：同步 Cookies 变化
- **当** HTTP 客户端的 cookies 发生变化
- **那么** 系统必须更新当前活跃账号的 cookies
- **并且** 系统必须将变化持久化到本地存储

### 需求：更新用户资料

系统必须支持更新账号的用户资料信息。

#### 场景：手动更新用户名和头像
- **当** 调用 `updateActiveProfile` 方法
- **那么** 系统必须更新当前活跃账号的用户名和头像
- **并且** 系统必须将变化持久化到本地存储
- **并且** 系统必须发送账号更新通知

#### 场景：从服务器刷新用户资料
- **当** 调用 `refreshActiveAccountProfile` 方法
- **那么** 系统必须调用用户信息 API
- **并且** 如果成功，系统必须更新用户名和头像
- **并且** 如果失败，系统必须保持原有资料不变

### 需求：移除账号

系统必须支持删除已保存的账号。

#### 场景：删除指定账号
- **当** 用户删除某个账号
- **那么** 系统必须从账号列表中移除该账号
- **并且** 系统必须持久化更新后的账号列表

#### 场景：删除当前活跃账号
- **当** 删除的账号是当前活跃账号
- **那么** 系统必须清除活跃账号状态
- **并且** 如果还有其他账号，系统必须自动激活最近活跃的账号
- **并且** 如果没有其他账号，系统必须清除所有 cookies
- **并且** 系统必须发送账号更新通知

#### 场景：删除所有账号
- **当** 用户执行退出登录操作
- **那么** 系统必须清空所有保存的账号
- **并且** 系统必须清除活跃账号状态
- **并且** 系统必须清除所有 HTTP cookies
- **并且** 系统必须发送账号更新通知

### 需求：处理认证失败

系统必须处理 token 过期或失效的情况。

#### 场景：检测到认证失败
- **当** API 请求返回认证失败错误（-101）
- **那么** 系统必须调用 `handleAuthenticationFailure` 方法
- **并且** 系统必须移除当前活跃账号
- **并且** 如果有其他账号，系统必须切换到其他账号
- **并且** 如果没有其他账号，用户必须重新登录

### 需求：账号列表管理

系统必须支持多账号管理功能。

#### 场景：获取所有账号列表
- **当** 请求账号列表
- **那么** 系统必须返回所有已保存的账号
- **并且** 账号列表必须按最后活跃时间降序排列

#### 场景：账号去重
- **当** 尝试注册已存在的账号（相同 mid）
- **那么** 系统必须更新现有账号而不是创建重复记录

### 需求：发送状态变更通知

系统必须在账号状态变化时通知应用其他部分。

#### 场景：账号变更时发送通知
- **当** 账号被添加、删除、切换或更新
- **那么** 系统必须通过 NotificationCenter 发送 `AccountManagerDidUpdate` 通知
- **并且** 通知对象必须是 AccountManager 实例

### 需求：存储数据模型

系统必须定义清晰的数据结构来存储账号信息。

#### 场景：Account 数据结构
- **当** 创建账号记录
- **那么** 账号必须包含以下字段：
  - `token`: LoginToken - 包含 mid、accessToken、refreshToken、expiresIn、expireDate
  - `profile`: Profile - 包含 mid、username、avatar
  - `cookies`: [StoredCookie] - HTTP cookies 列表
  - `lastActiveAt`: Date - 最后活跃时间

#### 场景：数据可编码性
- **当** 保存或加载账号数据
- **那么** Account、Profile 和 LoginToken 必须遵循 Codable 协议
- **并且** 系统必须能够成功编码和解码所有字段
