## 上下文

**当前状态**: 主 Tab Bar 包含 6 个标签（推荐、热门、排行榜、关注、收藏、我的），在 `BLTabBarViewController.swift` 的 `viewDidLoad()` 方法中顺序创建并添加。应用启动时从 UserDefaults 读取 `selectedIndex` 恢复用户最后选择的 Tab。

**背景**: 先前已移除直播 Tab，将 7 个 Tab 精简为 6 个。本次变更继续简化界面，移除发现型功能（推荐、热门、排行榜），专注于个性化内容管理（关注、收藏）。

**约束**:
- 必须保留被移除 Tab 对应的 ViewController 代码（FeedViewController、HotViewController、RankingViewController），为将来可能的其他入口保留可能性
- 必须正确处理旧的 `selectedIndex` 值（0-5），避免索引越界崩溃
- 不能影响关注、收藏、我的三个保留功能的正常运行

**利益相关者**: 
- 用户：习惯使用推荐、热门、排行榜功能的用户将无法从主界面访问这些功能
- 开发者：代码架构更简洁，主导航逻辑更清晰

## 目标 / 非目标

**目标：**
- 从主 Tab Bar 移除推荐、热门、排行榜三个 Tab 的 UI 入口
- 将主导航从 6 个 Tab 精简为 3 个 Tab（关注、收藏、我的）
- 正确处理 `selectedIndex` 的索引迁移，避免崩溃或错误导航
- 保持代码整洁，不删除底层 ViewController 实现

**非目标：**
- ❌ 不删除 FeedViewController、HotViewController、RankingViewController 的代码文件
- ❌ 不修改这三个 ViewController 的内部实现
- ❌ 不迁移或删除用户的任何本地数据（历史记录、缓存等）
- ❌ 不添加新的导航入口或功能

## 决策

### 决策 1: 直接删除 Tab 创建代码，保留 ViewController 文件

**选择**: 在 `BLTabBarViewController.swift` 中直接删除创建 FeedViewController、HotViewController、RankingViewController 的代码（约 9 行），但不删除这些文件本身。

**理由**: 
- 这些 ViewController 的功能完整且独立，可能在未来作为其他入口（如搜索结果、个人中心子页面）被重新使用
- 删除代码比注释代码更清晰，避免代码中出现大量注释干扰可读性
- 如需恢复功能，只需在适当位置重新实例化这些 ViewController

**替代方案考虑过但放弃**:
- ❌ 方案 A: 注释掉创建代码 → 会留下过多注释代码污染主文件，且编译器不检查注释代码的正确性
- ❌ 方案 B: 删除 ViewController 文件 → 违反"保留底层代码"的约束，且将来恢复需要从 git 历史找回

### 决策 2: 添加 selectedIndex 边界检查和映射逻辑

**选择**: 在 `viewDidLoad()` 中读取 `selectedIndex` 后，添加边界检查逻辑：
```swift
var savedIndex = UserDefaults.standard.integer(forKey: selectedIndexKey)
// 处理旧索引：0-2（推荐/热门/排行榜）映射到 0（关注）
// 处理旧索引：3-5（关注/收藏/我的）映射到 0-2
if savedIndex >= 0 && savedIndex <= 2 {
    savedIndex = 0  // 旧的推荐/热门/排行榜 → 关注
} else if savedIndex >= 3 && savedIndex <= 5 {
    savedIndex = savedIndex - 3  // 关注(3→0), 收藏(4→1), 我的(5→2)
} else {
    savedIndex = 0  // 无效索引 → 关注
}
selectedIndex = savedIndex
```

**理由**:
- 用户可能在上次使用时选择了推荐/热门/排行榜 Tab，保存的索引为 0-2，直接使用会导向关注/收藏/我的，造成混乱
- 用户可能选择了关注/收藏/我的 Tab（旧索引 3-5），需要映射到新索引 0-2 以保持用户习惯
- 超出范围的索引（如旧版本的直播 Tab 索引）应有明确的 fallback 逻辑

**替代方案考虑过但放弃**:
- ❌ 方案 A: 直接使用 `selectedIndex = min(savedIndex, 2)` → 无法正确映射旧的关注/收藏/我的索引（3-5）
- ❌ 方案 B: 清空 UserDefaults 中的 selectedIndex → 每次升级都会重置用户选择，用户体验差
- ❌ 方案 C: 复杂的版本号检测和迁移逻辑 → 过度设计，单次映射逻辑足够处理所有情况

### 决策 3: 不修改其他引用 Tab 索引的代码

**选择**: 不主动搜索和修改其他可能引用 Tab 索引的代码（如埋点、分析、快捷跳转逻辑）。

**理由**:
- 从代码结构看，没有明显的硬编码索引引用（如 `selectedIndex = 3`）
- 如果存在此类代码，运行时会自然触发边界检查逻辑，fallback 到关注 Tab，不会崩溃
- 过度搜索和修改可能引入新的 bug，采用"保守变更"策略更安全

**风险**: 如果存在隐藏的硬编码索引引用，可能导致跳转到错误的 Tab。缓解措施：在测试阶段关注导航跳转功能的正确性。

## 风险 / 权衡

### 风险 1: 用户习惯中断

**风险**: 依赖推荐/热门/排行榜功能的用户将无法从主界面访问这些功能，可能造成困惑或不满。

**缓解措施**: 
- 这是 BREAKING 变更，在 proposal 中已明确标识
- 如果用户反馈强烈，可以考虑在"我的"页面中添加到这些功能的入口，或通过搜索功能替代
- 底层代码保留，恢复功能的成本很低（只需重新添加 Tab 创建代码）

### 风险 2: 索引映射逻辑未覆盖所有边界情况

**风险**: 可能存在特殊的 `selectedIndex` 值（如负数、极大值）未被正确处理，导致崩溃或错误导航。

**缓解措施**:
- 使用 `else` 分支捕获所有未明确处理的情况，统一 fallback 到索引 0（关注）
- UITabBarController 自身会对 `selectedIndex` 进行边界检查，超出范围的值会被忽略
- 测试阶段手动修改 UserDefaults 值，验证各种边界情况的处理

### 风险 3: 隐藏的 Tab 索引硬编码引用

**风险**: 代码库中可能存在其他地方硬编码了 Tab 索引（如 `tabBarController.selectedIndex = 3` 跳转到关注），修改后会跳转到错误的 Tab。

**缓解措施**:
- 在应用设计阶段应避免硬编码索引，应使用 ViewController 类型判断或枚举
- 如发现此类问题，可通过 grep 搜索 `selectedIndex` 和具体索引数字定位并修复
- 测试阶段重点测试应用内的导航跳转功能（如通知点击、深度链接等）

## 权衡

### 权衡 1: 功能可访问性 vs. 界面简洁性

**权衡**: 移除 3 个 Tab 使界面更简洁聚焦，但牺牲了发现型功能的可访问性。

**选择**: 优先界面简洁性。理由是应用定位为"个人化内容管理工具"，用户主要需求是访问关注和收藏的内容，而非持续发现新内容。如有需要，可通过其他非主导航的入口提供发现功能。

### 权衡 2: 代码保留 vs. 代码清理

**权衡**: 保留 ViewController 代码会使代码库中存在"孤岛代码"（未被引用的模块），增加维护负担；但删除后恢复功能需要从 git 历史找回。

**选择**: 保留代码。理由是 Swift 项目的编译器会自动忽略未使用的代码（不会打包到最终 binary），存储成本几乎为零。且保留代码让将来的扩展更灵活（如在设置中添加"实验功能"开关）。
