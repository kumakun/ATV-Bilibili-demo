# follow-feed

## 目的

iOS 端更新页面的动态 Feed 功能，展示用户关注 UP 主发布的视频动态，支持筛选、刷新和分页加载。

## 新增需求

### 需求：系统必须以多列网格展示更新动态

系统必须将更新动态页面中的视频内容以多列网格方式展示，以提高浏览效率并充分利用横向空间。

#### 场景：在 iPhone 上显示更新动态

- **当** 用户在 iPhone 上打开更新页面
- **那么** 系统必须以 2 列网格展示视频动态卡片
- **那么** 系统必须保证每个卡片仍可显示封面、标题、UP 主信息和核心播放信息

#### 场景：在更宽环境中显示更新动态

- **当** 更新页面运行在 iPad 或其他更宽的可用宽度环境中
- **那么** 系统必须使用多于 2 列的网格展示视频动态卡片
- **那么** 系统必须根据可用宽度自适应增加列数
- **那么** 系统禁止退化为与 iPhone 相同的固定 2 列布局，除非可用宽度已缩小到无法满足卡片最小可读宽度

#### 场景：宽度变化时保持可读性

- **当** 设备方向、窗口尺寸或容器宽度发生变化
- **那么** 系统必须重新计算网格列数
- **那么** 系统必须优先保证卡片内容可读，而不是盲目增加列数
- **那么** 布局变化后分页加载、刷新和点击跳转行为必须保持正常

### 需求：系统必须展示关注动态列表

系统必须在更新页面展示用户关注的 UP 主发布的视频动态列表，包括视频封面、标题、UP 主信息、发布时间和播放数据。

#### 场景：用户打开更新页面

- **当** 用户打开更新页面
- **那么** 系统必须自动加载并展示第一页的关注动态列表
- **那么** 每个动态卡片必须显示视频封面、标题、UP 主头像、UP 主名称、发布时间、播放量、弹幕数

#### 场景：没有关注动态

- **当** 用户没有关注任何 UP 主或关注的 UP 主没有发布动态
- **那么** 系统必须显示空状态提示："暂无关注动态"

#### 场景：加载失败

- **当** 动态数据加载失败（如网络错误）
- **那么** 系统必须显示错误提示信息
- **那么** 系统必须提供"重试"按钮允许用户重新加载

### 需求：系统必须支持下拉刷新

系统必须支持用户通过下拉手势刷新关注动态列表，获取最新的动态内容。

#### 场景：用户下拉刷新

- **当** 用户在动态列表顶部下拉
- **那么** 系统必须显示刷新指示器
- **那么** 系统必须重新加载第一页的动态数据
- **那么** 刷新完成后，系统必须更新列表内容并隐藏刷新指示器

#### 场景：刷新期间用户松手

- **当** 用户在刷新过程中松开手指
- **那么** 系统必须继续完成刷新操作
- **那么** 系统禁止中断刷新过程

### 需求：系统必须支持分页加载

系统必须在用户滚动到列表底部时自动加载更多动态内容，实现无限滚动体验。

#### 场景：滚动到底部加载更多

- **当** 用户滚动到当前列表的最后一个动态项
- **那么** 系统必须自动触发加载下一页数据
- **那么** 系统必须在列表底部显示"加载中"指示器

#### 场景：没有更多动态

- **当** 服务器返回的动态数量少于请求数量或标记为没有更多数据
- **那么** 系统必须停止加载更多动态
- **那么** 系统禁止在滚动到底部时继续发起请求

#### 场景：防止重复加载

- **当** 系统正在加载更多数据时
- **那么** 系统禁止同时发起多个加载请求
- **那么** 系统必须等待当前请求完成后才能发起下一个请求

### 需求：系统必须支持筛选器切换

系统必须提供筛选器允许用户在"全部"、"视频"、"直播"之间切换，以过滤显示的动态类型。

#### 场景：切换筛选器选项

- **当** 用户点击筛选器中的不同选项（全部/视频/直播）
- **那么** 系统必须更新筛选器的选中状态
- **那么** 系统必须显示对应筛选条件的动态列表

#### 场景：筛选器的初始状态

- **当** 用户首次打开关注页面
- **那么** 筛选器必须默认选中"全部"选项

### 需求：系统必须支持视频详情导航

系统必须允许用户点击动态卡片跳转到对应的视频详情页面。

#### 场景：点击动态卡片

- **当** 用户点击任意动态卡片
- **那么** 系统必须导航到该视频的详情页面
- **那么** 系统必须传递正确的视频 ID（aid、cid、epid）参数

#### 场景：点击 UP 主头像或名称

- **当** 用户点击动态卡片中的 UP 主头像或名称
- **那么** 系统应当导航到该 UP 主的个人空间页面（如已实现）

### 需求：系统必须过滤非视频动态

系统必须只展示包含视频内容的动态，禁止展示纯图文、转发等非视频类型的动态。

#### 场景：API 返回混合类型动态

- **当** API 返回的动态列表中包含图文、转发等非视频内容
- **那么** 系统必须只展示包含 archive（投稿视频）或 pgc（番剧影视）类型的动态
- **那么** 系统禁止展示其他类型的动态内容

### 需求：系统必须显示加载状态

系统必须在加载数据期间向用户提供清晰的加载状态反馈。

#### 场景：首次加载

- **当** 用户首次打开关注页面且列表为空
- **那么** 系统必须在页面中央显示加载指示器和"加载中..."文本

#### 场景：加载更多

- **当** 用户滚动到底部触发加载更多
- **那么** 系统必须在列表底部显示小型加载指示器

#### 场景：下拉刷新

- **当** 用户下拉刷新列表
- **那么** 系统必须在列表顶部显示系统标准的刷新指示器

### 需求：系统必须使用 offset-based 分页

系统必须使用服务器返回的 offset 字符串进行分页，而不是传统的页码数字。

#### 场景：初始加载请求

- **当** 系统首次加载动态列表
- **那么** 系统必须使用空字符串作为 offset 参数

#### 场景：加载下一页请求

- **当** 系统加载更多动态时
- **那么** 系统必须使用上一次响应中返回的 offset 值作为请求参数

#### 场景：刷新列表重置 offset

- **当** 用户下拉刷新列表
- **那么** 系统必须将 offset 重置为空字符串
- **那么** 系统必须清空现有列表并重新加载第一页数据
