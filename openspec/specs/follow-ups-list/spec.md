# follow-ups-list

## 目的

关注 UP 主列表功能，在 iOS 端展示用户关注的所有 UP 主，支持分页加载、网格布局显示，并支持导航到 UP 主的个人空间页面。

## 需求

### 需求：显示关注列表
系统必须以列表形式显示用户关注的所有 UP 主。每个 UP 主必须显示头像、用户名和个人签名。

#### 场景：成功加载关注列表
- **当** 用户打开关注 UP 主列表页面
- **那么** 系统必须从服务器获取关注列表数据
- **并且** 显示包含头像、用户名和签名的 UP 主卡片

#### 场景：加载失败
- **当** 获取关注列表失败
- **那么** 系统必须显示错误提示信息
- **并且** 允许用户重试

#### 场景：空列表
- **当** 用户没有关注任何 UP 主
- **那么** 系统必须显示空状态提示

### 需求：分页加载
系统必须支持分页加载关注列表，每页加载固定数量的 UP 主数据。

#### 场景：首次加载
- **当** 用户打开关注列表
- **那么** 系统必须加载第一页数据（最多 40 个 UP 主）

#### 场景：滚动加载更多
- **当** 用户滚动到列表底部
- **并且** 还有更多数据未加载
- **那么** 系统必须自动加载下一页数据

#### 场景：已加载全部数据
- **当** 用户滚动到列表底部
- **并且** 已加载所有关注的 UP 主
- **那么** 系统禁止继续请求更多数据

### 需求：防止重复请求
系统必须防止在上一次请求未完成时发起新的数据请求。

#### 场景：请求进行中
- **当** 系统正在加载某一页数据
- **并且** 用户触发加载更多操作
- **那么** 系统必须忽略新的请求
- **并且** 等待当前请求完成

### 需求：显示 UP 主头像
系统必须为每个 UP 主显示圆形头像，头像必须从网络加载并进行适当优化。

#### 场景：加载 UP 主头像
- **当** 显示 UP 主卡片
- **那么** 系统必须加载并显示该 UP 主的头像
- **并且** 头像必须以圆形形式显示
- **并且** 使用图片降采样优化内存

#### 场景：头像加载失败
- **当** UP 主头像加载失败
- **那么** 系统必须显示默认头像占位符

### 需求：显示 UP 主信息
系统必须显示 UP 主的用户名和个人签名。

#### 场景：显示完整信息
- **当** UP 主有用户名和签名
- **那么** 系统必须显示用户名在上方
- **并且** 显示个人签名在下方

#### 场景：签名过长
- **当** UP 主签名文本过长
- **那么** 系统必须截断或省略显示

### 需求：导航到 UP 主空间
系统必须支持用户点击 UP 主卡片后导航到该 UP 主的个人空间页面。

#### 场景：点击 UP 主卡片
- **当** 用户点击某个 UP 主的卡片
- **那么** 系统必须打开该 UP 主的个人空间页面
- **并且** 传递 UP 主的 mid 参数

### 需求：网格布局显示
系统必须使用网格布局展示关注列表，并且必须根据设备类型与可用宽度采用不同的列数策略，以保证浏览效率和卡片内容可读性。

#### 场景：在 iPhone 上显示关注列表
- **当** 在 iPhone 设备上显示关注列表
- **那么** 系统必须使用 2 列网格展示关注 UP 主卡片
- **并且** 系统必须保持每个卡片可显示头像、用户名和签名摘要

#### 场景：在 iPad 上显示关注列表
- **当** 在 iPad 设备上显示关注列表
- **那么** 系统必须使用多列网格展示关注 UP 主卡片
- **并且** 网格列数必须根据可用宽度自适应增加
- **并且** 系统禁止退化为与 iPhone 相同的 2 列移动端布局（除非窗口宽度已缩小到无法满足卡片最小可读宽度）

#### 场景：在 iPad 宽屏下提高内容密度
- **当** 关注列表在 iPad 的常规宽度或更宽布局中显示
- **那么** 系统必须在同一屏幕中展示多于 iPhone 2 列布局的卡片数量
- **并且** 保持卡片间距与内容排版稳定，不因增加列数造成主要信息不可读

#### 场景：窄宽度环境下保持可读性
- **当** 关注列表所在窗口宽度缩小（例如 iPad 分屏或尺寸变化）
- **那么** 系统必须重新计算网格列数
- **并且** 优先保证卡片内容可读，而不是盲目维持更高列数
- **并且** 布局调整后必须保持分页加载与点击导航行为正常
